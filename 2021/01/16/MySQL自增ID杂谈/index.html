MySQL杂谈

背景：基于Innodb存储引擎。

MySQL杂谈

自增ID

自增ID的优点

  1. 如果建表时定义了主键，Innodb会选择主键作为聚集索引。如果没有显式定义主键，Innodb会选择第一不包含null值的唯一索引作为主键索引。如果也没有这样的唯一索引，Innodb会选择内置6bytes长的ROWID作为隐含的聚集索引；
  2. 数据记录被存于主索引的叶子节点，同一叶节点（页）的数据记录按主键值顺序存放。每当有新的记录插入时，会根据新纪录主键将其放入合适的节点和位置；
  3. 使用自增主键，每次插入新的记录都会顺序添加至当前索引节点的后续位置，与底层数据结构（B+树）逻辑相符；
  4. 如果采用非自增主键，新记录可能要被放入现有索引节点中间的某个位置，MySQL为了存储数据可能需要不断移动数据。有的目标页可能已经被写回磁盘，此时又需要读取它们，增加IO开销。此外，频繁的移动和分页操作会产生大量的碎片，影响数据库读写性能。

自增ID的缺点

  1. 需要手动插入指定ID的记录时操作复杂。此外插入指定Id时很可能会破坏连续性；
  2. 当需要合并表的时候，两个表的ID很可能发生冲突；
  3. 分布式架构中，也可能存在ID冲突。

自增ID溢出

MySQL中，Int整型的范围如下：

类型 最小值 最大值 存储大小
Int 2^31^ 2^31^ 4 bytes
unsigned Int 0 2^32^ 4 bytes

当自增id达到最大值，继续插入数据会报一个主键冲突异常。

简单的处理方式：将Int改为BigInt（8字节）。

那么如何在线修改数据类型呢？

  1. 使用MySQL提供的修改功能

    在MySQL5.5之前，该功能通过拷贝一个临时表实现。执行ALTER后，会新建一个带有新结构的临时表，然后将原表的数据全部拷贝至临时表，然后Rename，完成创建操作。整个过程中原表是可读的，但是不能写。5.6后，在以上过程中，针对绝大部分操作，原表可读也可写。

    但是采用ALTER修改数据类型，在整个更改过程表是无法进行更新操作的（DELETE, UPDATE, DELETE）。

  3. 改变从库表的结构，然后进行主从切换。

在实际场景中，很难遇到自增主键达到最大值的场景，因为单表达到一定数量级后就会进行分库分表，这时就不能依赖每个表的自增ID来唯一标识这些数据了，需要提供一个全局唯一的ID生成策略。

为什么采用B+树

与其他数据结构的比较

为什么不采用二叉树：当数据是依次递增的时候，建立的二叉树实际上就是一个链表。

为什么不采用哈希表：虽然查找单个记录很快，但是不适用于范围查询和排序。

为什么不采用红黑树：红黑树是一种特化的平衡二叉树，在进行插入和删除操作时通过特定操作保持二叉查找树的平衡。当数据库数据流很大时，索引的体积也会很大，内存可能存不下，因此需要从磁盘上进行读写。如果树高太高，则读写磁盘的次数（IO交互）会很多，影响性能。

为什么不采用B树：B树的查找分为两步。首先查找节点，由于B树通常是在磁盘上存储的，因此这步需要磁盘IO操作；然后查找关键字，当找到某个节点后将该节点读入内存中然后通过顺序或者折半查找来查找关键字。若没有找到关键字，则需要判断大小来找到合适的分支继续查找。

B树算法的执行时间主要由读、写磁盘来决定，因此一次IO操作应该尽可能多的读写信息。因此B树的节点规模一般以一个磁盘页为单位。

而B+树相对B树而言，相邻的叶子节点间用一个双向链表链接，并且非叶子节点只存储索引，从而存放更多索引。B+树相对B树进行范围查找只需要查找一个节点然后进行遍历，而B树则需要获取所有节点。此外B+树内部节点并没有指向关键字具体信息的指针，内部节点更小，一个节点能容纳的索引更多。

总结

1.单个节点存储越多的元素，树高越低，自然在整个过程中与磁盘的IO交互就越少；但是无限增加树的路数并不可取，因为文件系统和数据库的索引都是存在硬盘中的，数据量如果过大，不一定能一次性加载到内存中。

2.相对B树，所有的查询最终都会找到叶子节点，性能稳定；高度可控。

3.所有叶子节点通过双向链表相连，范围查找方便。
